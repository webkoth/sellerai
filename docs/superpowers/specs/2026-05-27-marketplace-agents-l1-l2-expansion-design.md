## Контекст

Заказчик SellerAI опубликовал вакансию менеджера маркетплейса. На неё откликнулись 27 кандидатов с резюме, отражающими реальный рынок специализаций: дизайнеры инфографики, менеджеры внутренней рекламы, операционные директора, специалисты по внешнему трафику/блогерам, закупщики из Китая, compliance-эксперты, AI-креаторы и т.п.

Существующее рабочее пространство `sai_kotelnikovartifact` уже содержит:
- **5 агентов:** `ads-optimizer`, `content-optimizer`, `marketplace-analyst`, `review-manager`, `social-media-content-transformer`
- **22 skill** (`wb-*`, `ozon-*`, `ym-*`, `mp-accounting`, `profitability-report`, etc.)
- **17 команд** (`/audit`-эквиваленты по узким темам: `/seo-audit`, `/funnel`, `/weekly-report`, etc.)
- **4 MCP-сервера:** wb-mcp, ozon-mcp, ym-mcp, avito-mcp

**Выявленные пробелы** относительно реальных ролей рынка (из анализа резюме):
- 🔴 Operations Director (стратегический слой)
- 🔴 External Traffic Manager (блогеры, внешка)
- 🔴 Infographic Designer (8 кандидатов → массовый спрос)
- 🔴 AI Visual Generator (нейрофотосессии)
- 🟡 China Procurement (1688/Taobao) — оставлено на Phase 3
- 🟡 Compliance Officer (Честный знак) — оставлено на Phase 3

## Цели

1. **Phase 1:** закрыть два критичных пробела — `operations-director` (L1) и `external-traffic-manager` (L2)
2. **Phase 2 (roadmap):** подготовить базу для `infographic-designer` и `ai-visual-generator` через skill-stubs
3. Сохранить принцип "внутренний инструмент оператора" — без cron, без авто-роутизации
4. Не плодить избыточные команды/MCP — переиспользовать существующее

## 1. Архитектура

### Иерархия после Phase 1

```
┌──────────────────────────────────────────────────────────────┐
│  L1 — operations-director (NEW)                              │
│  Оркестратор полного аудита, стратегических задач,           │
│  приоритизации проблем кабинета. Вызывает L2 через Agent.    │
└──────────────────────────────────────────────────────────────┘
                  │
   ┌──────────────┼──────────────┬──────────────┬──────────────┐
   ▼              ▼              ▼              ▼              ▼
ads-optimizer  content-      marketplace-   review-       external-
(существ.)     optimizer     analyst        manager       traffic-mgr
               (существ.)    (существ.)     (существ.)    (NEW)
```

### Data flow: комплексный аудит

```
1. Оператор → operations-director (через Agent tool, "Аудит магазина X")
2. operations-director планирует подзадачи:
   ├─ "Анализ рекламы за 30 дней"          → ads-optimizer
   ├─ "SEO-аудит топ-10 карточек"          → content-optimizer
   ├─ "Финансовая аналитика и продажи"     → marketplace-analyst
   └─ "Свежие отзывы, негатив, рейтинг"    → review-manager
3. Все 4 запускаются ПАРАЛЛЕЛЬНО (один message с 4 Agent tool calls)
4. operations-director агрегирует в стратегический отчёт:
   ├─ 🔴 Критичные проблемы (топ-3)
   ├─ 🟡 Точки роста (топ-5)
   ├─ Сводный KPI-дашборд
   └─ План действий на 30 дней
5. → отчёт оператору
```

### Data flow: оценка КП от блогера

```
1. Оператор → external-traffic-manager (текст КП + артикул товара)
2. Агент:
   ├─ Извлекает условия из КП
   ├─ Запрашивает marketplace-analyst → юнит-эк товара
   ├─ Рассчитывает CPV, CPO, безубыточность
   └─ Формирует вердикт: 🟢/🟡/🔴 + контрпредложение
3. → ответ оператору + запись в bloggers.jsonl
```

### Ключевое правило архитектуры

**L1 НЕ делает работу L2 сам.** Operations Director вызывается **только** для:
- Комплексного аудита (>2 направлений)
- Стратегических задач (план роста, юнит-эк портфеля, приоритизация)
- Разрешения конфликтов между L2 (например: ставка vs маржа)

Для узких задач (только реклама / только отзывы) оператор вызывает L2 напрямую.

## 2. Agent: `operations-director`

### Frontmatter

```yaml
---
name: operations-director
description: L1-агент для комплексных стратегических задач по магазину. Используйте для аудитов кабинета, планирования роста, приоритизации проблем, разрешения конфликтов между направлениями. НЕ используйте для узких задач (только реклама / только отзывы) — обращайтесь к профильному агенту напрямую. Триггеры — "проведи аудит", "стратегия роста", "что в первую очередь чинить", "план развития кабинета".
tools: ["Agent", "Read", "Bash", "Grep"]
color: gold
---
```

**Обоснование `tools`:**
- `Agent` — для делегирования L2-агентам (ключевое)
- `Read`, `Bash`, `Grep` — для чтения файлов отчётов, данных в `data/`, сохранения итогов
- MCP-инструменты **намеренно не подключены** — за L1 стоят L2

### Компетенции

| Область | Что делает | На основе резюме |
|---------|-----------|------------------|
| Комплексный аудит | Запускает все 4 L2 параллельно, склеивает в один отчёт с приоритетами | Андрей (Geo Home), Юлия 7+ лет, "готовый отдел продаж" |
| Стратегия роста | Юнит-экономика портфеля, план запуска новинок, ABC по SKU | Андрей (260 млн), Анна |
| Приоритизация | Ранжирование проблем по `потенциал × срочность × лёгкость` (ICE-МП) | Стас, Ольга |
| Разрешение конфликтов | Между рекомендациями L2 (повысить ставку vs снизить цену) | роль опер. дира |
| Финансовая сводка | Связка с `profitability-report` skill и `mp-accounting` | Анна, WB Fin кейс |

### Skills

- Существующие: `wb-unit-economics`, `wb-pricing-strategy`, `wb-supply-chain`, `mp-accounting`, `profitability-report`, `demand-forecasting`
- **Новый:** `marketplace-operations-director` (методология ICE-МП, шаблон отчёта, правила конфликтов)

### Алгоритм работы (системная инструкция агента)

```
1. Прими запрос. Определи тип:
   - "audit"      → запусти все 4 L2 параллельно
   - "strategy"   → ads-optimizer + marketplace-analyst, потом план
   - "priority"   → marketplace-analyst (диагностика) → сам ранжируешь
   - "conflict"   → собери позиции конфликтующих L2, разреши через юнит-эк
2. Для каждого делегирования сформируй ПОЛНЫЙ prompt субагенту:
   - Цель ("Нужен SEO-аудит топ-10 SKU для общего аудита магазина X")
   - Контекст (бизнес-модель, ниша, цели клиента)
   - Формат ожидаемого результата (краткая сводка, не сырые данные)
   - Ограничения (период анализа, какие метрики критичны)
3. Дождись всех результатов. Склей в формат "Стратегический отчёт".
4. Никогда не дублируй работу L2 — только агрегируй и расставляй приоритеты.
```

### Формат итогового отчёта

```markdown
# Стратегический аудит: [Магазин], [Дата]

## ⚡ TL;DR
- Главная проблема: [одна фраза]
- Главная возможность: [одна фраза]
- Эффект от рекомендаций: +X₽/мес при затратах Y₽

## 🔴 Критично (требует действия в 7 дней)
| # | Проблема | Источник | Эффект | Действие | Ответственный L2 |
|---|----------|----------|--------|----------|------------------|
| 1 | ROI рекламы −15% по 3 кампаниям | ads-optimizer | −180k/мес | Пауза + рекалибровка | ads-optimizer |

## 🟡 Точки роста (4 недели)
…

## 📊 KPI-дашборд
| Метрика | Текущее | Бенчмарк | Дельта |

## 🗺️ План действий на 30 дней
Неделя 1: …

## ⚠️ Конфликты направлений
[если есть: чьи рекомендации противоречат и почему выбран Х]
```

### Hooks (минимальные)
Никаких `PreToolUse` (агент не работает с MCP). Опционально один `PostToolUse` после Write — напомнить оператору о сохранении отчёта в `data/audits/`.

## 3. Agent: `external-traffic-manager`

### Frontmatter

```yaml
---
name: external-traffic-manager
description: Агент для работы с внешним трафиком — оценка предложений от блогеров/менеджеров внешки, расчёт экономики интеграций, написание ТЗ на интеграцию, контроль публикаций. Используйте для входящих КП от блогеров, планирования закупки рекламы у инфлюенсеров, сравнения внутренней и внешней рекламы по CPO. Триггеры — "оцени блогера", "посчитай внешку", "стоит ли брать рекламу у", "ТЗ на интеграцию".
tools: ["Agent", "Read", "Bash", "Grep", "WebFetch"]
color: cyan
---
```

**Обоснование `tools`:**
- `Agent` — запрос юнит-эк у `marketplace-analyst`
- `WebFetch` — публичная проверка канала блогера (TG/Insta линк)
- `Read`/`Bash`/`Grep` — работа с `data/external-traffic/`

### Компетенции

| Область | Что делает | Источник идеи |
|---------|-----------|---------------|
| Оценка КП от блогеров | Извлекает условия, проверяет на манипуляции | "Менеджер блогеров", Кристина |
| Юнит-экономика интеграции | CPV, CPO, прогноз заказов, точка безубыточности | КП с тарифами 10/20/30 интеграций |
| Сравнение каналов | Внешка vs внутренняя реклама по CPO | связка с `ads-optimizer` |
| ТЗ на интеграцию | Reels-сценарий, ключ для выкупа, шаблон отзыва | пакет "Reels+Stories+выкуп" |
| Проверка на накрутку | ER, динамика подписчиков, чаты активности | "проверка на накрутку" |
| Контрпредложение | Переговорная позиция: цена, условия, гарантии | сила менеджеров внешки |

### Skills

- Существующие: `wb-advertising` (бенчмарки CPO внутренней), `wb-unit-economics`
- **Новый:** `external-traffic-economics` (формулы CPV/CPO внешки, бенчмарки)
- **Новый:** `blogger-vetting-checklist` (проверка на накрутку)

### Алгоритм оценки КП

```
ВХОД: текст КП + ссылка/артикул товара клиента + (опционально) линк на канал

1. ИЗВЛЕЧЕНИЕ ИЗ КП:
   - Цена за интеграцию / пакет (₽)
   - Формат (Reels / Stories / пост / бартер)
   - Заявленный охват (просмотры/подписчики)
   - Что входит: выкуп? отзыв? подъём в поиске?
   - Гарантии

2. ДЕЛЕГАЦИЯ marketplace-analyst (Agent tool):
   "Дай юнит-экономику товара X: средний чек, маржа,
    CPO внутр. рекламы за 30 дней, ДРР, выкупаемость"

3. РАСЧЁТЫ:
   - CPV = цена / охват (норма: 0.5–2 ₽)
   - Клики = охват × CTR-внешки (бенчмарк 0.5–1.5%)
   - Заказы = клики × CR (с учётом холодного трафика, 0.5×CR-внутр)
   - CPO-внешки = цена / заказы
   - Безубыточность = цена / маржа

4. ВЕРДИКТ:
   🟢 БРАТЬ      если CPO-внешки < CPO-внутр × 1.3
   🟡 ПЕРЕГОВОРЫ если CPO-внешки в коридоре 1.3–2.0
   🔴 ОТКАЗАТЬ   если CPO-внешки > CPO-внутр × 2.0
   ИСКЛЮЧЕНИЕ: новый бренд + охват >100k ЦА → выше для brand awareness

5. (ОПЦИОНАЛЬНО) ПРОВЕРКА КАНАЛА через WebFetch:
   - Динамика подписчиков (скачки = накрутка)
   - ER = реакции / просмотры (норма 3–8%)
   - Релевантность аудитории по постам

6. ВЫХОД: вердикт + контрпредложение + ТЗ (если 🟢/🟡)
```

### Формат вердикта

```markdown
# Оценка КП: [Имя/канал], [Дата]

## Вердикт: 🟢 БРАТЬ / 🟡 ПЕРЕГОВОРЫ / 🔴 ОТКАЗАТЬ

## Цифры
| Метрика | Заявлено | Расчёт | Норма | Статус |
|---------|----------|--------|-------|--------|
| Цена | 5 000 ₽ | — | — | — |
| Охват | 50 000 | — | — | — |
| CPV | — | 0.10 ₽ | 0.5–2 | 🟢 |
| Прогноз заказов | — | 12–18 | — | — |
| CPO внешки | — | 280–420 ₽ | — | — |
| CPO внутр. | — | 350 ₽ | — | — |
| Безубыточность | — | 14 заказов | — | — |

## Риски
- [из проверки канала / условий КП]

## Контрпредложение
[цена, формат, гарантии]

## ТЗ на интеграцию (если 🟢/🟡)
- Ключевой запрос для выкупа: …

```

## 4. Новые skills

### 4.1 Phase 1 — три полноценных skill

#### `marketplace-operations-director`

Содержимое `SKILL.md`:
1. **Фреймворк приоритизации ICE-МП**: Impact × Confidence × Ease, Score = I·C·E
2. **Шаблон стратегического отчёта** (TL;DR / 🔴/🟡 / KPI / 30-60-90 план)
3. **Разрешение конфликтов** между направлениями:
   - "Ставка vs Маржа" → ставка ≤ ДРР_лимит × средний чек / клики
   - "Скидка vs Прибыль" → юнит-эк до/после
   - "Поставка vs Оборачиваемость" → формула безопасного остатка
4. **Чек-лист комплексного аудита** (что собрать у каждого L2, какие данные нужны от клиента)
5. **Примеры кейсов** (из резюме): запуск ремней (Стас), Geo Home 20→80 SKU (Андрей), Подарочные наборы ТОП5 (Павел)

#### `external-traffic-economics`

Содержимое `SKILL.md`:
1. **Формулы:** CPV, CTR-внешки, CR-внешки, CPO-внешки, безубыточность
2. **Бенчмарки по нишам** (одежда, дом, косметика, электроника, +18) — CTR/CR/охваты
3. **Пороги вердикта:** 🟢 ≤1.3 / 🟡 1.3–2.0 / 🔴 >2.0 от CPO-внутр; исключение для brand awareness
4. **Шаблоны:** ТЗ на Reels, скрипт сторис (проблема→решение→CTA), шаблон отзыва, контрпредложение
5. **Красные флаги в КП:** гарантии без статистики, только бартер без метрик, цены >2× медианы, отсутствие стат канала

#### `blogger-vetting-checklist`

Содержимое `SKILL.md`:
1. **Быстрая проверка (5 мин):** динамика подписчиков (TGStat/Telemetr), ER 3–8%, разброс просмотров по последним 10 постам, релевантность комментариев
2. **Глубокая проверка (30 мин):** охваты публикаций vs заявленных, география аудитории, чаты активности (Adminbot, ACTIV), публичные интеграции
3. **Критерии отказа:** ER <1% при >10k подписчиков, скачок подписчиков за месяц, нет аватарок у >30%, комменты — эмодзи
4. **Инструменты:** TGStat (бесплатные функции), Telemetr.me, WebFetch публичного канала

## 5. Slash-команды

### Phase 1 — две новых

#### `/audit [артикул | бренд | "all"] [период=30d]`

`.claude/commands/audit.md`:
```markdown
---
description: Комплексный стратегический аудит магазина через Operations Director
argument-hint: "[артикул | бренд | all] [период=30d]"

Запусти комплексный аудит для $ARGUMENTS.

Шаги:
1. Распарси аргументы: первый — цель (артикул/бренд/all), второй — период.
2. Вызови агента `operations-director` через Agent tool с prompt'ом:
   "Проведи комплексный аудит [target] за [period]. Используй методологию ICE-МП из skill marketplace-operations-director. Запусти все 4 L2-агента параллельно: ads-optimizer, content-optimizer, marketplace-analyst, review-manager. Сформируй стратегический отчёт. Сохрани в data/audits/YYYY-MM-DD-[target].md".
```

#### `/external-deal [артикул товара]` (+ текст КП в следующем сообщении)

`.claude/commands/external-deal.md`:

```markdown
---
description: Оценка коммерческого предложения от блогера/менеджера внешки
```

## 7. Тестирование и валидация Phase 1

### Подход
Не пишем unit-тесты для агентов (это markdown-prompt'ы). Вместо этого — **5 smoke-сценариев**, прогоняемых вручную после реализации. Результаты фиксируются в `data/audits/smoke-results.md`.

### Smoke 1: operations-director базовый

```
Команда: /audit all 30d
Ожидание:
✓ запускает 4 L2 параллельно (один message с 4 Agent tool calls)
✓ не вызывает MCP-инструменты сам
✓ каждый L2 возвращает структурированный отчёт
✓ L1 агрегирует в формат "Стратегический отчёт"
✓ TL;DR — 3 строки максимум
✓ KPI-дашборд с дельтой к бенчмарку
✓ файл сохранён в data/audits/YYYY-MM-DD-all.md
```

### Smoke 2: operations-director разрешение конфликта

```
Подготовка: товар с маржой 5%, ads-optimizer рекомендует +ставка,
marketplace-analyst видит убыточность
Команда: /audit [артикул]
Ожидание:
✓ выявляет конфликт
✓ применяет правило из marketplace-operations-director (ставка ≤ ДРР × чек / клики)
✓ выдаёт решение со ссылкой на юнит-эк
✓ секция "⚠️ Конфликты направлений" присутствует
```

### Smoke 3: external-traffic-manager оценка КП

```
Команда: /external-deal [артикул]
Передать: текст реального КП ("Менеджер блогеров" из 27 резюме)
Ожидание:
✓ извлекает условия (10 интеграций по 500₽, формат и т.д.)
✓ запрашивает marketplace-analyst юнит-эк
✓ считает CPV, CPO, безубыточность
✓ сравнивает CPO-внешки с CPO-внутр
✓ вердикт 🟢/🟡/🔴 с обоснованием
✓ при 🟢/🟡 — ТЗ с ключом для выкупа + сценарий + шаблон отзыва
✓ блогер добавлен в bloggers.jsonl
```

### Smoke 4: external-traffic-manager отказ по красному флагу

```
Передать: КП с признаками накрутки (гарантии без статистики, только бартер)
Ожидание:
✓ идентификация красных флагов из blogger-vetting-checklist
✓ вердикт 🔴
✓ перечислены конкретные риски
✓ контрпредложение содержит требование статистики
```

### Smoke 5: изоляция контекстов

```
Команда: аудит магазина с 1000+ SKU через operations-director
Ожидание:
✓ L1 НЕ читает сырые данные SKU (это работа marketplace-analyst)
✓ L1 получает только агрегированный отчёт от L2
✓ контекст L1 остаётся компактным
```
