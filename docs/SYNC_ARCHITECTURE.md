# `/sync` — архитектура зеркалирования WB → Ozon / Яндекс.Маркет

> **Статус:** дизайн утверждён (2026-06-03). Реализация — следующая итерация.
> **Мастер:** Wildberries. Направление — **одностороннее**: WB → Ozon + ЯМ.
> Клиент ведёт ассортимент вручную только на WB; Ozon и ЯМ — зеркала.

---

## 0. Принцип

- **WB — единственный источник правды (SSOT)** по составу, контенту, ценам и остаткам.
- **Никакого «универсального матчера»** поверх трёх разных моделей данных (это и ломало прошлые попытки). Вместо него — **адаптер на каждую площадку** (WB→Ozon, WB→ЯМ) + **явные таблицы маппинга** в `data/mappings/`.
- **`/sync` — оркестратор** над тремя раздельными слоями с разной частотой:

```
                 WB (SSOT): карточка · цена · остаток
                                  │
              /sync (оркестратор, по умолчанию полное зеркало)
                                  │
        ┌─────────────────┬───────┴────────┬─────────────────┐
   Слой 1: Карточка   Слой 2: Цена     Слой 3: Остаток
   (редко, тяжёлый)   (часто)          (очень часто, точно)
        │                                       │
   WB→Ozon adapter                         WB→ЯМ adapter
```

| Слой | Частота | Что делает |
|------|---------|------------|
| 1. Карточка/контент | при новом товаре / правке контента | категория, атрибуты, адаптация текста, фото |
| 2. Цены | часто | WB-цена × коэффициент площадки (комиссии) |
| 3. Остатки | очень часто | точное совпадение с WB |

---

## 1. SSOT-карты (`data/mappings/`)

| Файл | Назначение | Статус |
|------|-----------|--------|
| `category-map.json` | WB-предмет → Ozon (category+type) / ЯМ (marketCategoryId) + чёрный список ювелирки | ✅ заполнен |
| `pricing.json` | ставки комиссий + коэффициенты `k` (режим A) + guardrails | ✅ заполнен |
| `identity-map.json` | WB barcode ↔ Ozon offer_id ↔ ЯМ offerId | ⏳ схема (наполняется при reconcile) |
| `attribute-map.json` | WB-характеристика → атрибут площадки (+ словарные значения) | ⬜ Phase 0 (TODO) |

**Принцип карт:** всё, что раньше было хардкодом в Python/TS, выносится в эти файлы. Маппинг **явный и проверяемый человеком**, без тихих fallback.

## 2. Идентичность товара

- **Канонический ключ = баркод WB** (`sizes[].skus[]`). Для этого продавца товар в основном уникален (1 шт, метеориты) → связь **1 WB-баркод = 1 оффер** на каждой площадке.
- **`offerId` новых офферов** на Ozon и ЯМ = баркод WB (единая схема).
- **Уже залитые офферы** линкуются по факту: WB↔Ozon чаще по `vendorCode` (старая схема), WB↔ЯМ по `barcode`. `identity-map.json` фиксирует фактический id каждой площадки → апдейты бьют в нужный оффер, без дублей.
- **Фаза reconcile** (часть `/sync`): тянет WB-карточки + Ozon offers + ЯМ offers, матчит по баркоду (fallback vendorCode для Ozon), проставляет статусы `linked / missing / conflict`. `missing` → кандидат на заливку; `conflict` → ревью.

## 3. Слой 1 — Карточка / контент

### 3.1 Категория (боль №1 — решена)
- Маппинг **только из `category-map.json`**, по WB `subjectName`.
- **Ювелирные категории — чёрный список, не используются никогда** (товар — бижутерия/сувениры ручной работы, не ювелирка с клеймом/ГИИС ДМДК).
- **Незамапленный предмет → СТОП + алерт** оператору. Не лить кривое.
- Категории между площадками **не обязаны совпадать** (Ozon обереги/минералы → бижутерия `17027899`; ЯМ → сувенирная ветка).
- Исключённые предметы (`excluded:true`, напр. «Открытки») — не зеркалируются.

### 3.2 Атрибуты
- Из `attribute-map.json` (Phase 0): WB-характеристика → атрибут площадки + `dictionary_value_id`.
- Обязательные атрибуты заполняются по (категория+тип); недостающие → флаг, товар на ревью (не лить с дырами).
- Источник требований: `ozon_get_category_attributes(category_id, type_id)` — тянется один раз на тип, кэшируется в `attribute-map`.

### 3.3 Контент (режим: **A по умолчанию + SEO по флагу**)
- **A (дефолт):** название/описание под лимиты площадки (Ozon имя ≤200 / описание ≤6000; ЯМ ≤2000) с **умной обрезкой по предложению**; фото как есть (WB CDN); бренд/страна/габариты 1:1 (см→мм для Ozon, см+кг для ЯМ).
- **`/sync --seo`:** название/описание **переписываются под ключи и стиль площадки** (LLM на карточку) — для приоритетных товаров.

### 3.4 Фото
- Переносятся WB CDN-ссылки как есть (webp принимается обеими площадками). Конвертация в jpg/png — опция при проблемах загрузки.

## 4. Слой 2 — Цены (боль №2 — решена)

**Режим A — полное сохранение маржи** (`pricing.json`):

```
k_площадка   = (1 − ставка_WB) / (1 − ставка_площадки)
цена         = round_up_10( wb_finalPrice × k )
old_price    = round_up_10( wb_basePrice × k )   // только если basePrice > finalPrice
```

- `wb_finalPrice` = `discountedPrice` (эталон, не пересчёт), `wb_basePrice` = цена до скидки.
- Ставки: WB ~32% против Ozon ~53% / ЯМ ~52% → `k ≈ ×1.43–1.48` (цены на зеркалах выше WB на ~40%; для уникального товара приемлемо).
- **Guardrails:** округление вверх до 10 ₽; `old_price` только при реальной скидке (нет «отрицательной скидки»); Ozon `min_price` = цена × 0.95; потолок наценки не задан (режим A).
- Цены пересинхронизируются при каждом `/sync` (не «один раз и забыли»).

---

## 5. Слой 3 — Остатки

- **Точное совпадение с WB.** `WB stock (FBS, по баркоду) → Ozon/ЯМ stock` через `identity-map`.
- ЯМ: `updatedAt` = текущее время (требование «не старше суток»).
- Атомарность: остаток обновляется по фактическому `offerId` площадки из identity-map.
- Это самый частый и лёгкий слой; в `/sync` его можно гонять отдельно (`--only stocks`).

## 6. Команда `/sync` — пошаговый процесс

### Phase 0 — построение карт (один раз, с ревью оператора)
1. Pull WB-предметов + Ozon `get_category_attributes` по целевым типам + ЯМ-категории.
2. Собрать `attribute-map.json`; дозаполнить TODO в `category-map`/`pricing` (часы, серьги).
3. Оператор подтверждает спорные места.

### `/sync` (рабочий прогон)
```
/sync [--mp ozon|ym|all] [--only cards|prices|stocks] [--seo] [--dry-run]
```
1. **Reconcile** — обновить `identity-map` (что залито, что `missing`).
2. **Слой 1 (cards)** — для новых/изменённых: категория по карте (нет → стоп) → атрибуты → контент (A или `--seo`) → **preview** → импорт → дождаться модерации → записать в identity-map.
3. **Слой 2 (prices)** — пересчитать по `pricing.json` → **preview БЫЛО→СТАЛО** → апдейт.
4. **Слой 3 (stocks)** — WB → площадка, точное совпадение.
5. **Отчёт** — что синхронизировано, что `missing`/`conflict`/на ревью, ошибки модерации.

**По умолчанию (без флагов) — полное зеркало (cards+prices+stocks, обе площадки).**

## 7. Guardrails и подтверждения

- 🚫 **Незамапленная категория → СТОП**, не лить.
- 🚫 **Ювелирные категории — никогда.**
- 🔒 **Изменения цен/остатков/карточек — через preview БЫЛО→СТАЛО**, применение по `confirm=true` (как в MCP-инструментах).
- 🔁 **Остатки — точное совпадение** с WB.
- 📋 **Лишние офферы на Ozon/ЯМ, которых нет в WB** — в отчёт, не трогаем без подтверждения.
- ⏳ **Модерация Ozon/ЯМ асинхронна** — карточки появляются с задержкой; `/sync` опрашивает статус.
